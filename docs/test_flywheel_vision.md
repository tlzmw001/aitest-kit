# AI 测试飞轮 — 愿景与设计

## 我们要解决什么问题

企业项目中，测试用例的生成和维护面临三个核心痛点：

1. **文档缺失**：设计文档不完整甚至不存在，测试人员只能靠猜或读代码
2. **用例质量低**：手写用例容易被实现"锚定"，验证的是"代码做了什么"而非"代码该做什么"
3. **经验不积累**：踩过的坑没有沉淀机制，同样的错误反复出现

## 飞轮架构

四步闭环，每转一圈质量提升一级：

```
需求/代码变更
     │
     ▼
┌─────────────┐
│  /doc-gen   │  从源码 + 现有文档 → 面向测试的设计文档
└──────┬──────┘
       │
       ▼
┌──────────────────┐
│ /knowledge-build │  设计文档 → 三层知识库（L0 架构 / L1 模块 / L2 变更）
└──────┬───────────┘
       │
       ▼
┌──────────────┐
│ /test-design │  知识库 + 测试规范 + 历史用例 → 新用例 + 覆盖变更 + mismatch
└──────┬───────┘
       │
       ▼
  用户评审 / 执行
       │
       ▼  （发现问题时）
┌─────────────┐
│  /test-fix  │  修正用例 + 记录陷阱到 TEST_SPEC → 下轮 test-design 自检
└─────────────┘
```

飞轮的动力来自两个正反馈回路：
- **覆盖回写**：test-design 生成用例后，自动刷新知识库的"已有测试覆盖"章节，下次生成时避免重复
- **陷阱积累**：test-fix 把错误模式写入 TEST_SPEC 的"已知陷阱"，test-design 每次生成前读取并自检

## 核心设计决策

### 两轮生成策略

| | 第一轮：业务用例 | 第二轮：边界用例 |
|---|---|---|
| 信息源 | 知识库（L1 业务规则 + L2 变更） | 源代码实现 |
| 禁止 | 读源代码 | 修改第一轮的预期结果 |
| 产出 | business.md | boundary.md + mismatch.md |
| 预期结果 | 锚定业务规格 | 锚定代码实际行为 |

第一轮不看代码，是为了让预期结果来自"系统该做什么"而非"系统做了什么"。第二轮读代码发现的规格偏差不改用例，而是记录为 mismatch — 这是最高价值的产出，因为它暴露了需求与实现的裂缝。

### 知识库三层结构

- **L0 — 系统架构索引**（≤80 行）：一段话描述 + 模块索引表，路由到具体 L1
- **L1 — 模块功能契约**（≤150 行/文件）：输入输出、业务规则、错误场景、可观测状态
- **L2 — 需求变更**（≤100 行/文件）：变更 delta、影响分析、测试重点

知识库采用灰盒视角：包含 API 契约、配置项、存储 Key 格式、错误码，但不包含函数名、行号等源码细节。

### 用例可执行性要求

用例必须写到可直接执行的粒度：
- 输入：完整请求 JSON，不能只写关键参数片段
- 前置条件：具体构造方式（Redis 命令、配置文件内容、实验参数）
- 测试步骤：`POST /api/recommend，body 为上述 JSON`，不能写"发送推荐请求"

### 用例与知识库的链接

- 知识库侧：引用覆盖维度（"已覆盖：基本路由映射、policyId 兜底"），不引用具体编号
- 用例侧：标注 `related: L1/scene_routing`，指向知识库模块
- 变更时只改用例文件，知识库覆盖摘要由 skill 自动刷新

### 从用例到执行

```
Markdown 用例（唯一数据源）
     │
     ▼  codegen（确定性模板映射）
pytest .py 文件（编译产物，不手动编辑）
     │
     ▼
CI 执行 → 测试报告
```

用例按可自动化程度分三级：
- **L-Auto**：后端 API 调用，完全自动执行
- **L-Semi**：请求自动但断言需人工（检查日志/监控等副作用）
- **L-Manual**：前端操作/视觉验证，Markdown 作为手工测试指引

质量关口前移到 Markdown 层：人只 review 用例 diff，不 review 生成的 .py。

## Skill 体系

| Skill | 用途 | 调用方式 |
|-------|------|---------|
| /doc-gen | 从源码生成面向测试的设计文档 | `/doc-gen <source_dir> [doc_dir] <output_dir>` |
| /doc-review | 审查设计文档完整性（7 维度评分） | `/doc-review [doc_dir]` |
| /knowledge-build | 构建/更新三层测试知识库 | `/knowledge-build <doc_dir> [knowledge_dir]` |
| /test-design | 两轮生成测试用例 | `/test-design <模块名>` |
| /test-fix | 修正用例 + 记录陷阱 | `/test-fix <用例ID> <错误描述>` |

### 产品形态

Skill 层（Claude Code 编排）+ Python 工具库（aitest-kit，确定性逻辑）。两层解耦：
- Skill 负责 AI 理解代码 → 生成方案 → 编排流程
- aitest-kit 负责 HTTP/gRPC 请求执行、断言、Redis 数据准备、报告生成

底层工具库可复用，未来可脱离 Claude Code 接 API 变独立产品。

## 质量保障

### TEST_SPEC 测试规范

所有用例生成的"宪法"，包含：
- 用例格式（含可执行性要求）、编号规则、优先级定义
- 两轮生成策略的具体约束
- 质量红线 Q1-Q10（覆盖编号唯一性、预期结果来源、可执行性等）
- 排除场景 / 关注场景 / 已知陷阱（三个迭代章节，随使用积累）

### 质量红线摘要

| 编号 | 要求 |
|------|------|
| Q1 | 编号全局唯一，不重复不回收 |
| Q2 | 优先级必须标注 |
| Q3 | 第一轮预期结果只能来自知识库 |
| Q4 | 知识库没说的行为标 TBD |
| Q5 | 每条用例标注 related 指向知识库 |
| Q6 | 不生成排除列表中的场景 |
| Q7 | 关注列表中的场景必须全部覆盖 |
| Q8 | 输入必须是完整可发送的请求体 |
| Q9 | 前置条件必须写出具体构造方式 |
| Q10 | 测试步骤必须写到可直接执行 |
