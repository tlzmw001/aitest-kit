# TEST SPEC

本文件记录当前目标系统的测试编号规则、质量红线、排除/关注场景和已知陷阱。新项目初始化后，先补齐模块缩写表，再开始生成 Markdown 用例。

## 模块缩写对照表

| 模块名 | TC 缩写 | 说明 |
|--------|---------|------|
| `{module_name}` | `{MOD}` | 初始化占位；开始设计用例前替换或删除 |

## 优先级定义

| 优先级 | 含义 |
|--------|------|
| P0 | 核心链路、发布阻断、数据安全或高风险回归 |
| P1 | 重要业务规则、常见边界和主要异常路径 |
| P2 | 低频边界、可观测性检查、兼容性或补充覆盖 |

## 质量红线 Q1-Q10

| 编号 | 规则 |
|------|------|
| Q1 | 每条用例必须有唯一且符合 `TC-{缩写}-{序号}` 的编号 |
| Q2 | 共享配置中的 HTTP 基础请求体必须是合法 JSON，不使用 `{{placeholder}}` |
| Q3 | 场景变量必须写成 `key：value` 条目，不能用大段自然语言替代 |
| Q4 | 前置条件必须可执行，写清配置、测试数据、管理 API 或外部依赖状态 |
| Q5 | 断言必须指向可观察结果，不能写“结果正确”“功能正常”等空泛表述 |
| Q6 | 不猜测系统中间产物固定值；不可预知值使用关系断言、范围断言或人工标记 |
| Q7 | `[manual]` 和 `[!可行性存疑: 原因]` 必须写在独立标记字段 |
| Q8 | 生成 pytest 前优先修 Markdown、profile、fixture、helper 或 project_config |
| Q9 | generated pytest 是编译产物，不作为长期手写源文件维护 |
| Q10 | 确认待测系统 bug 时记录到 `test_workspace/results/`，不通过 skip 或放宽断言掩盖 |
