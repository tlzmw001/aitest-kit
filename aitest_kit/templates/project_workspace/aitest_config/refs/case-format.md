# 用例格式：共享配置 + 精简用例

> 本文档定义测试用例的 Markdown 输出格式。
> 引用方：test-design skill（生成用例）、codegen（解析 MD 生成 pytest）

## 设计原则

- 每个模块的用例文件顶部定义**共享配置**，包含所有用例共用的部分
- 每条用例只写与共享配置不同的**场景变量**和**断言**
- 目标：人类 review 时一眼看到"这条用例测什么"，不被重复信息干扰

## 文件结构

```markdown
# {模块名} 业务测试用例

> 生成方式：test-design skill
> 关联知识库：L1/xxx
> 生成日期：YYYY-MM-DD

---

## 共享配置

**接口**：`POST /api/v1/resources/evaluate` / `gRPC package.Service/Method`

**基础请求体（HTTP）**：

```json
{
  "request_id": "req_default",
  "user_id": "user_default",
  "resource_id": "resource_default",
  "attributes": {}
}
```

（必须是 `json.loads` 可解析的合法 JSON；变化字段填合法默认值。场景变量中的“请求覆盖”用于设计 review；确定性 codegen 的真实请求覆盖以 `codegen_profile` 的 `request_overrides` 为准）

**基础请求体（gRPC）**：
（完整 protobuf message 字段，如有两种接口）

**标准前置**：
- 所有用例共用的环境准备步骤（配置准备、测试数据初始化、外部依赖状态等）

**通用断言**：`response.code == 0`

**变量定义**：
- `raw` = `response.items[0].raw_value`（系统计算得到的中间值，不可预知）
- `derived` = `response.items[0].derived_value`
- `clamp(x, 0, 1)` = `max(0, min(1, x))`

---

## 一、{场景分组名}

### TC-XXX-001：{一句话描述}
- **优先级**：P1
- **场景变量**：
  - 协议：HTTP
  - 请求覆盖：`{"user_id": "u_xxx_001"}`
  - 前置操作：通过管理 API 写入测试资源 `{"resource_id":"resource_a", "status":"active"}`
- **断言**：`derived == clamp(1.5 * raw + 0.1, 0, 1)`

### TC-XXX-002：{一句话描述}
- **优先级**：P1
- **场景变量**：请求覆盖：`{"user_id": "u_xxx_002"}`
- **断言**：`derived == raw`（规则未命中时保持原值）
- **标记**：`[manual]`

## 覆盖变更

| 知识库文档 | 新增覆盖 | 仍未覆盖 |
|-----------|---------|---------|
| L1/xxx | 维度 A、维度 B | 维度 C |
```

## 共享配置区字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 接口 | 是 | 列出模块暴露的所有接口类型（HTTP endpoint + gRPC method） |
| 基础请求体 | 是 | 完整的请求 JSON/protobuf，每种接口一份；HTTP JSON 必须是合法 JSON，不使用 `{{placeholder}}` |
| 标准前置 | 是 | 所有用例共用的环境准备（配置、测试数据、外部依赖状态等） |
| 通用断言 | 是 | 每条用例都要检查的断言 |
| 变量定义 | 按需 | 关系断言中的变量缩写，减少每条用例的重复 |

## 每条用例字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 优先级 | 是 | P0/P1/P2，异常类型追加标注（如 `P1 / 异常`） |
| 场景变量 | 是 | 与共享配置的差异，必须写成 `key：value` 条目列表（见下方书写约束）。注意：这里的“请求覆盖”用于设计 review；确定性 codegen 的真实请求覆盖以 `codegen_profile` 的 `request_overrides` 为准 |
| 断言 | 是 | 可程序化的断言表达式，引用变量定义中的缩写 |
| 标记 | 否 | `[manual]`、`[!可行性存疑: 原因]` 等标记，独立一行，不内联到其他字段 |

## 场景变量书写约束

场景变量必须写成 `key：value` 条目列表，禁止写成自然语言段落。每个条目一行，key 用中文冒号分隔。

单条目可简写为行内格式：

```markdown
- **场景变量**：请求覆盖：`{"scene_name": "ad", "device": "pc"}`
```

多条目换行：

```markdown
- **场景变量**：
  - 请求覆盖：`{"scene_name": "ad", "device": "pc"}`
  - 前置操作：`PUT /api/v1/test-data/u_xxx {"mode": "fallback"}`
```

key 名称不做全局限定，由各项目按实际需要自行约定，保持同一模块内一致即可。

## 标记字段

当用例存在特殊状态时，使用独立的"标记"字段，不要把标记内联到场景变量或断言中：

```markdown
### TC-XXX-009：某场景描述
- **优先级**：P2 / 异常
- **场景变量**：...
- **断言**：...
- **标记**：`[manual]`、`[!可行性存疑: 需要测试环境能区分 X 和 Y]`
```

已定义标记：
- `[manual]` — 无法自动化执行，需人工验证
- `[!可行性存疑: 原因]` — 用例设计合理但执行可行性待确认

## 旧版完整格式（兼容）

已有用例可保留以下格式，不强制迁移：

```markdown
### TC-{模块}-{序号}：{一句话描述}
- **关联**：L1/xxx 或 L2/xxxx
- **优先级**：P0 / P1 / P2
- **类型**：业务 / 边界 / 异常
- **前置条件**：环境状态、数据准备
- **输入**：完整请求体（JSON 格式）
- **测试步骤**：
  1. 动作一
  2. 动作二
- **预期结果**：可断言的具体值
```
