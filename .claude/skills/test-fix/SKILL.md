# 测试用例修正

修正用例 `$case_id`，错误描述：`$error_description`。

## 执行流程

### 第一步：定位用例

1. 从 `$case_id` 中提取模块缩写（如 TC-ROUTE-013 → ROUTE）
2. 查 `test_workspace/knowledge/TEST_SPEC.md` 的模块缩写对照表，确定模块名
3. 在以下位置搜索该用例：
   - `test_workspace/cases/{模块名}/business.md`
   - `test_workspace/cases/{模块名}/boundary.md`
   - `cases/old-cases/` 下的文件
4. 读取该用例的完整内容和上下文（前后用例、关联的知识库文档）

### 第二步：分析错误

1. 读取用例关联的知识库文档（L1/L2）
2. 如果错误涉及代码行为，读取相关源代码确认实际行为
3. 判断错误类型：
   - **预期结果错误** — 预期值与实际行为不符
   - **前置条件不可行** — 无法按描述构造测试环境
   - **输入不完整** — 请求体缺字段或格式错误
   - **测试步骤遗漏** — 关键操作未写出
   - **逻辑错误** — 用例设计本身有问题（如验证了不可能的场景）
4. 向用户确认修正方案（如果错误原因不明确）

### 第三步：修正用例

1. 在用例文件中修正该用例
2. 修正后的用例必须满足 TEST_SPEC 质量红线（Q1-Q10）
3. 如果修正影响了覆盖变更清单，同步更新

### 第四步：记录陷阱

在 `test_workspace/knowledge/TEST_SPEC.md` 的"已知陷阱"章节追加一条记录：

```markdown
### 陷阱-{序号}：{一句话描述错误模式}
- **错误写法**：原来怎么写的（具体到可对照）
- **正确做法**：应该怎么写
- **原因**：为什么原来的写法是错的
- **适用范围**：哪类用例需要注意这个问题
```

陷阱序号从已有最大序号 +1 开始。

### 第五步：完成输出

向用户输出：

```markdown
## 用例修正摘要

修正用例：$case_id
错误类型：（预期结果错误 / 前置条件不可行 / ...）
修正内容：（改了什么）
新增陷阱：陷阱-{序号}（一句话描述）
```

## 约束

- 只修正用户指定的用例，不扩大修改范围
- 修正预期结果时，必须有依据（知识库规则或代码行为），不能猜测
- 陷阱记录必须抽象到可复用的模式，不能只描述单个用例的具体问题
- 如果错误是知识库本身的问题（规则描述不准确），在陷阱中标注，但不修改知识库（知识库修改需用户单独确认）
