# 测试知识库构建指令

## 你的角色

你是一个新入职的测试人员。你不能阅读任何源码，只能从开发文档中获取信息。

## 信息来源（严格限定）

- 只读 `test_workspace/docs/prd/` 目录下的文档
- 禁止读取项目中的任何 .py / .proto / .yaml / .json / .tsv 等代码和配置文件
- 禁止读取 `docs/`、`coupon_system/`、`ab_experiment_sdk/` 等目录
- 禁止读取 `.claude/plans/` 下的 plan 文件
- 每条信息必须能溯源到文档原文，文档里没有的就标 `[?]`

## 输出位置

`test_workspace/knowledge/` 目录下，结构：

```
test_workspace/knowledge/
├── L0_system_architecture.md    # 系统架构索引
├── L1/                          # 模块功能文档（每模块一个文件）
└── L2/                          # 需求变更文档（每迭代一个文件）
```

## 分层说明

### L0 — 系统架构（路由索引）
- 控制在 80 行以内
- 一段话描述系统功能
- 服务拓扑
- Pipeline 阶段概览
- 模块索引表：模块名 | 说明 | L1 路径 | 相关 L2 路径
- 不包含具体业务规则

### L1 — 模块功能（可测试契约）
- 每个文件控制在 150 行以内
- 功能差异大的拆成独立文件
- 反映当前最新全貌
- 使用固定模板（见下方）

### L2 — 需求变更
- 每个文件控制在 100 行以内
- 描述变更 delta 和测试重点
- 使用固定模板（见下方）

## L1 模板

```markdown
# 模块名
一句话功能描述

## 输入
从哪来、什么格式、关键字段

## 输出
到哪去、什么格式、关键字段

## 业务规则
1. （精确、可测试的自然语言描述）
2. ...

## 错误场景
- 条件 → 错误码/行为

## 可观测状态
- Redis Key / 响应字段 / 日志等可验证的外部状态

## 关联 L2
- [变更名](../L2/xxx.md) — 一句话说明
```

## L2 模板

```markdown
## 变更概述
一句话描述

## 影响的 L1 模块
- 模块名 — 具体影响点

## 新增/变更规则
1. ...

## 影响分析
- 哪些已有功能可能被影响，需要回归

## 测试重点
- 新逻辑的关键路径
- 容易出问题的边界
```

## 信息边界（灰盒）

可以包含：API 接口定义、请求/响应结构、配置含义、Redis Key（如果文档提到）、错误码、业务算法公式

不包含：源码函数名、行号、变量名、内部实现细节

## 开始

1. 读 `test_workspace/docs/prd/` 下所有文档
2. 根据文档内容决定 L1 需要哪些模块文件
3. 按模板生成 L0、L1、L2
4. 文档中没有的信息一律标注 `[?]`，不猜测
