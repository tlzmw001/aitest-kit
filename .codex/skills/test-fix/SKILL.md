# Test Fix

用于修正单条或少量测试用例，并判断问题是否需要反向修正相关 skill。

## 适用场景

- 用户指出某条测试用例有误
- 用例执行失败，需要修正前置条件、输入、步骤或预期结果
- 需要把踩坑经验补充进 `TEST_SPEC.md`

## 输入

- 用例编号，如 `TC-ROUTE-013`
- 错误描述

## 执行流程

### 1. 定位用例

根据用例编号：

1. 提取模块缩写
2. 在 `test_workspace/knowledge/TEST_SPEC.md` 中查模块缩写对照
3. 在以下位置查找目标用例：
   - `test_workspace/cases/{模块名}/business.md`
   - `test_workspace/cases/{模块名}/boundary.md`
   - `test_workspace/cases/old-cases/`

### 2. 分析错误

读取：

- 目标用例
- 相邻上下文
- 关联的 L1/L2 文档

如果问题涉及真实行为，再读取相关代码确认。

将错误归类为：

- 预期结果错误
- 前置条件不可行
- 输入不完整
- 测试步骤遗漏
- 逻辑错误

### 3. 修正用例

只修正用户指定范围内的用例，并确保修正后满足 `TEST_SPEC` 质量红线。

如果覆盖变更清单受影响，同步更新。

### 4. 记录陷阱

在 `test_workspace/knowledge/TEST_SPEC.md` 的“已知陷阱”章节追加一条：

```markdown
### 陷阱-{序号}：{一句话描述}
- **错误写法**：...
- **正确做法**：...
- **原因**：...
- **适用范围**：...
```

陷阱要抽象成可复用模式，而不是只描述单个 case。

### 5. 做 skill 级根因分析

判断这是：

- 个别用例问题
- 还是 skill 指令导致的系统性问题

出现以下情况时，应更新相关 skill：

- 同类错误反复出现
- 原有 skill 步骤缺少必要检查
- 原有 skill 没要求读取关键输入
- 原有断言策略会系统性生成错误用例

### 6. 更新对应 skill

如果是系统性问题：

1. 判断责任归属：
   - `test-design`
   - `knowledge-build`
   - `doc-review`
2. 读取对应的 Codex skill：`.codex/skills/<skill>/SKILL.md`
3. 调整步骤、边界或质量约束
4. 在完成摘要中明确说明更新了哪个 skill、为什么更新

## 完成输出

默认按以下结构汇总：

```markdown
## 用例修正摘要

修正用例：...
错误类型：...
修正内容：...
新增陷阱：...
Skill 更新：无 / 已更新 xxx
```

## 质量约束

- 只修正目标用例，不扩大范围
- 修正预期结果必须有依据，不能猜
- 如果问题根源是知识库不准确，要明确指出，但不要越权改知识库定义
- 先修正具体 case，再判断是否需要反向更新 skill
