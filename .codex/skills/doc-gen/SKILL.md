# Doc Gen

用于从源码、接口定义和现有文档中提取设计信息，生成面向测试的设计文档。

## 适用场景

- 项目缺少系统设计文档
- 文档不完整，测试团队无法据此构建知识库
- 需要从代码补文档，但又希望输出适合测试使用的结构化材料

## 输入与默认输出

- 源码目录：用户指定
- 现有文档目录：可选
- 输出目录：用户指定
- 模块过滤：可选，只分析指定模块

默认输出文件：

- `_overview.md`
- `_data_flow.md`
- `module_<name>.md`
- `_discrepancies.md`，仅当现有文档目录存在时生成

## 信息来源优先级

按以下优先级提取信息：

1. API 定义：`.proto`、路由文件、OpenAPI
2. 配置文件与配置加载代码
3. 入口文件：`main.py`、`app.py` 等
4. 业务代码
5. 数据文件：`.tsv`、`.csv`
6. 现有文档

## 代码阅读策略

- 先读签名和结构，再按需深入实现
- 重点深入：
  - 条件分支
  - try/except
  - 外部依赖调用
  - logging
- 跳过：
  - 测试文件
  - 构建工具
  - 迁移脚本
  - 仅有注释价值的 TODO

## 执行流程

### 1. 结构侦察

扫描源码目录，识别：

- 入口文件
- API 定义
- 配置文件
- 数据文件
- 业务代码

如果用户给了模块过滤，只对相关模块做深度分析。

### 2. 文档盘点

如果提供了现有文档目录：

- 读取文档并分类
- 建立每个模块的文档覆盖地图

### 3. 逐模块深度分析

对每个目标模块提取：

- 输入输出字段定义
- 业务规则
- 错误处理与降级
- 外部依赖
- 数据格式
- 可观测性
- 硬编码常量和潜在配置缺口
- 已定义但未使用的错误码、常量或配置

### 4. 端到端数据流追踪

从主编排流程出发，记录：

- 输入结构
- 处理步骤
- 输出结构
- 异常分支
- 模块间字段变化

### 5. 交叉比对

如果提供了现有文档目录，逐模块识别三类差异：

- 文档有、代码无
- 代码有、文档无
- 文档与代码不一致

### 6. 生成文档

输出目录下的文档建议使用以下结构：

```text
output_dir/
├── _overview.md
├── _data_flow.md
├── module_<name>.md
└── _discrepancies.md
```

## 输出约束

### `_overview.md`

包含：

- 系统功能
- 服务拓扑
- 技术栈
- API 列表
- Pipeline 概览
- 模块列表
- 配置结构

### `module_<name>.md`

包含：

- 模块职责
- API 契约
- 业务规则
- 错误处理
- 外部依赖
- 数据存储
- 可观测性
- 实现参考

### `_data_flow.md`

包含：

- 主流程
- 异常分支
- 数据结构变化

### `_discrepancies.md`

包含：

- 文档有但代码未实现
- 代码有但文档未覆盖
- 描述不一致
- 已定义但未使用

## 来源标注

每条规则和错误场景尽量标注来源：

- `[代码]`
- `[文档]`
- `[差异]`

如果没有现有文档，统一按 `[代码]` 处理。

## 质量约束

- 只描述代码和契约实际表现，不猜测产品意图
- 规则必须精确到可测试
- 主体描述行为，不堆函数名和行号
- 如果启用了模块过滤，只更新目标模块相关文件
- 差异报告要能直接服务后续 `knowledge-build`
