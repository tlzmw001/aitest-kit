# Test Design

将 Claude 版本的 `test-design` 工作流迁移为 Codex skill 时，按本模板执行。

## 适用场景

- 用户要求为某个模块生成测试用例
- 用户要求基于新需求补充已有测试用例
- 用户要求根据知识库增量刷新 `test_workspace/cases/{模块名}/` 下的产物

## 输入与默认输出

- 目标模块：由用户指定
- 默认输出目录：`test_workspace/cases/{目标模块}/`

如果用户明确给出输出目录，使用用户指定目录；否则默认写入模块目录。

## 执行流程

### 1. 读取知识与上下文

按顺序读取以下内容：

1. `test_workspace/knowledge/TEST_SPEC.md`
2. `test_workspace/knowledge/L0_system_architecture.md`
3. 目标模块对应的 L1 文档
4. 关联的 L2 文档
5. 目标模块已有测试用例目录

提取这些信息：

- 用例模板与编号规则
- 模块缩写
- 优先级定义
- 排除场景与必测场景
- 已知陷阱
- 输入输出定义
- 业务规则、错误场景、可观测状态
- 已有测试覆盖与未覆盖维度

先搜索再写：创建新文件前，先检查目标目录下是否已有 `business.md`、`boundary.md`、`mismatch.md`。

### 2. 第一轮生成业务用例

这一轮只基于知识库生成，不从源码推测行为。

要求：

- 每条业务规则至少生成一条用例
- 每条错误场景至少生成一条异常用例
- L2 中的新增或变更规则必须覆盖
- 已在知识库中标记为已覆盖的维度默认不重复生成
- 知识库无法确认的预期结果写 `TBD-需确认`
- 输入字段基于文档中的完整请求结构；结构缺失时标注 `[!请求体待补全]`
- 前置条件必须写出可执行的构造方式
- 测试步骤必须写到可以直接执行的粒度

输出文件：`business.md`

文件头模板：

```markdown
# {模块名} 业务测试用例

> 生成方式：test-design skill
> 关联知识库：L1/xxx
> 生成日期：YYYY-MM-DD
```

单条用例模板：

```markdown
### TC-{模块缩写}-{序号}：{一句话描述}
- **关联**：L1/xxx 或 L2/xxxx
- **优先级**：P0 / P1 / P2
- **类型**：业务 / 异常
- **前置条件**：环境状态、数据准备、依赖服务状态
- **输入**：请求参数、测试数据（具体值）
- **测试步骤**：
  1. 动作一
  2. 动作二
- **预期结果**：可断言的具体值或状态变化
```

### 3. 第二轮生成边界用例

这一轮允许读取目标模块相关源码，用于补足边界条件和验证第一轮可行性。

重点查找：

- 降级逻辑
- 类型转换
- 空值与空集合处理
- 精度处理
- 知识库未覆盖的条件分支

要求：

- 不直接覆盖第一轮的知识库结论
- 发现知识库与实现不一致时，新建 `mismatch.md`
- 若代码可以补全 `[!请求体待补全]` 或 `TBD-需确认`，则在对应用例中补全并注明依据
- 对无法确认可执行性的用例追加 `[!可行性存疑: 原因]`

输出文件：`boundary.md`

Mismatch 模板：

```markdown
### MISMATCH-{序号}：{一句话描述}
- **关联**：L1/xxx 或 L2/xxxx
- **知识库描述**：知识库中的规则原文
- **实际实现**：代码中的实际行为
- **影响**：对用户/业务的影响判断
- **建议**：修代码 / 补文档 / 待产品确认
```

### 4. 维护覆盖变更

在 `business.md` 和 `boundary.md` 末尾追加覆盖变更清单：

```markdown
## 覆盖变更

| 知识库文档 | 新增覆盖 | 仍未覆盖 |
|-----------|---------|---------|
| L1/xxx | 维度 A、维度 B | 维度 C |
```

如果当前任务明确要求同步知识库，再更新 L1/L2 文档中的“已有测试覆盖”章节；否则只在生成结果中记录，不擅自改知识库。

### 5. 增量模式

当目标目录下已有用例文件时：

1. 读取已有内容
2. 找出已覆盖场景
3. 仅追加未覆盖用例
4. 编号从当前最大序号继续
5. 将新内容插入到覆盖变更清单之前

## 质量自检

输出前逐条检查：

- 预期结果是否可断言，而不是“成功”“正常”
- 输入是否为具体数据，而不是抽象描述
- 前置条件是否可复现
- 测试步骤是否完整且可执行
- 每条用例是否只验证一个关注点
- 用例是否独立于执行顺序
- 第一轮预期结果是否仅来自知识库
- 输入是否为完整请求体
- 前置条件是否写明构造方法
- 步骤是否包含接口路径、方法和断言表达式

不满足上述要求的用例，不要输出。

## 输出总结模板

完成后，向用户汇总：

```markdown
## 用例生成摘要

目标模块：{目标模块}
输出目录：{输出目录}

### 生成文件
| 文件 | 用例数 | 类型 |
|------|-------|------|
| business.md | N 条 | 业务 + 异常 |
| boundary.md | N 条 | 边界 |
| mismatch.md | N 条 | 规格偏差 |

### 覆盖变更
| 知识库文档 | 新增覆盖维度 | 仍未覆盖维度 |

### TBD 项
（列出所有预期结果为 TBD-需确认 的用例）

### 可行性存疑
（列出所有标注了 [!可行性存疑] 的用例）
```

如果用户进一步给出“排除场景”或“必测场景”反馈，再按需回写 `TEST_SPEC.md`。
