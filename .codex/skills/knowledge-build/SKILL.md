# Knowledge Build

用于从文档和外部契约文件中构建或更新测试知识库。

## 适用场景

- 首次为某个系统建立测试知识库
- 有新需求文档，需要增量更新已有知识库
- 需要把文档整理成 `L0/L1/L2` 的可测试契约形式

## 信息边界

你扮演灰盒测试人员：

- 可以读取：文档、`.proto`、OpenAPI、配置文件、数据文件、Schema
- 不读取：源码实现文件，如 `.py`、`.go`、`.ts`、`.java`

所有信息都必须能溯源到文档或可读契约文件；不能确认的内容统一标 `[?]`。

## 输入与默认输出

- 输入：文档目录
- 输出目录：用户指定；若未指定，默认使用文档目录同级的 `knowledge/`

目标结构：

```text
knowledge_dir/
├── L0_system_architecture.md
├── L1/
└── L2/
```

## 执行流程

### 1. 盘点文档

读取输入目录中的文档，并分类为：

- 系统总览
- 迭代变更
- 接入指南
- 历史测试文档
- 其他

### 2. 判断模式

检查输出目录：

- 没有 L0：初始化模式
- 已有 L0：增量更新模式

### 3. 初始化构建

初始化模式下：

1. 识别系统模块与迭代
2. 创建 `L0`、`L1`、`L2` 结构
3. 生成初始知识库
4. 未确认信息标 `[?]`

### 4. 增量更新

增量模式下：

1. 读取已有 `L0_system_architecture.md`
2. 对比新文档与已有 `L2`
3. 新迭代新建 `L2`
4. 补充信息更新已有 `L2`
5. 变更影响到的模块同步更新对应 `L1`
6. 发现新模块时，新建对应 `L1`

### 5. 强制校验 L0 一致性

每次增量更新都必须检查：

- Pipeline 顺序是否变化
- 版本号是否变化
- 服务拓扑是否变化
- 模块索引表是否需要更新
- 接口契约链接是否仍然正确

### 6. 输出摘要

完成后汇总：

- 初始化还是增量
- 读取了哪些文档
- 生成或修改了哪些文件
- 新增了哪些 `[?]`

## 分层要求

### L0

只放系统级索引：

- 系统概述
- 服务拓扑
- Pipeline 概览
- 模块索引表
- 接口契约链接

不在 L0 堆具体规则。

### L1

每个模块一个文件，描述当前全貌：

- 输入
- 输出
- 业务规则
- 错误场景
- 可观测状态
- 关联 L2

如果字段定义已经在 proto 或 OpenAPI 中，优先链接，不重复复制整份字段表。

### L2

每个需求变更一份，描述增量：

- 变更概述
- 影响的 L1 模块
- 新增或变更规则
- 影响分析
- 测试重点

## `[?]` 标注规则

按类型标注，便于后续补全：

- `[?行为未定义: ...]`
- `[?值未明确: ...]`
- `[?可观测性缺失: ...]`

文档没写清楚就标注，不猜。

## 风险标注

对于未处理或高风险异常路径，用 `[!风险]` 标记。

例如：

- Redis 异常未处理
- 下游服务失败会直接中断主流程
- 关键路径缺日志

## 质量约束

- 只基于文档和契约文件，不扩展到源码实现
- 规则必须精确、可测试、可追溯
- 增量更新只改受影响的文件
- 每个 L1 必须包含“可观测状态”
- 对关键信息缺失要明确指出，不要模糊带过
