# Doc Review

用于审查某个文档目录中的设计文档，判断它们是否足够完整、精确且适合测试团队使用。

## 适用场景

- 用户要求检查设计文档质量
- 用户想知道现有文档是否足够支撑测试设计
- 在进入 `knowledge-build` 或 `test-design` 前，需要先补齐文档缺口

## 输入与输出

- 输入：用户指定的文档目录
- 输出：面向测试视角的审查报告；默认直接在对话中汇总，不强制写文件

## 审查流程

### 1. 文档盘点

读取目标目录下的文档，并按用途分类：

- 系统总览
- 迭代变更
- 接入指南
- 技术规格
- 其他

先给出文档清单，再进入逐维度检查。

### 2. 按 7 个维度检查

每个维度给出 `PASS`、`PARTIAL` 或 `MISSING` 结论，并写明缺口。

#### API 契约完整性

检查：

- 请求字段是否列出字段名、类型、必填、默认值、取值范围
- 响应结构是否结构化定义
- 错误响应是否定义
- 是否至少有一个调用示例

#### 错误处理与降级

检查：

- 外部依赖不可用时的行为
- 超时与重试策略
- 降级或兜底逻辑
- 错误如何传播给调用方

#### 配置与数据格式

检查：

- 配置文件格式和结构示例
- 配置字段的名称、类型、含义、取值范围
- Redis 等存储的 key/value 格式

#### 业务规则精确度

检查：

- 规则是否精确到可以直接写测试断言
- 数值边界是否明确
- 多规则冲突时的优先级
- 公式是否显式给出

#### 跨文档一致性

检查：

- 字段命名是否一致
- 默认值是否冲突
- 总览与详细文档是否同步
- 迭代变更是否已回写到总览文档

#### 辅助/管理接口覆盖

检查：

- 初始化、查询、调试类接口是否有文档
- 每个接口是否有请求/响应定义
- 用途是否清晰

#### 精炼度与可读性

检查：

- 是否重复且表述不一致
- 关键信息是否使用表格或列表
- 单篇文档是否聚焦单一主题
- 是否便于快速定位信息

### 3. 输出审查报告

默认按以下结构输出：

```markdown
# 文档审查报告

## 文档盘点

## 维度评分
| 维度 | 评分 | 关键缺口 |

## 详细发现
- 缺什么
- 建议补在哪份文档
- 建议补成什么样

## 优先修复项
1. ...
```

## 质量约束

- 聚焦测试可用性，不做泛泛而谈的“文档写得不错”
- 对 `PARTIAL` 和 `MISSING` 必须写具体缺口
- 建议应落到具体文档和具体补充内容
- 不要求用户先改文档再继续，但要明确哪些缺口会阻塞后续知识库或测试设计
