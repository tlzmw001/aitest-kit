# L1 模块文档模板

> 本模板定义知识库 L1（模块功能文档）的标准结构。
> 引用方：knowledge-build skill（产出 L1 文档）、test-design skill（读取 L1 生成用例）

## 模板

```markdown
# 模块名
一句话功能描述

## 接口
（如有 HTTP/gRPC 测试入口，列出端点路径和 Schema 定义文件链接）
- HTTP 端点：`POST /api/v1/xxx`
- gRPC 端点：`package.Service/Method`
- 请求/响应完整字段定义：[proto 或 Schema 文件链接]

## 输入
从哪来、什么格式、关键字段（含透传的必填字段或其 Schema 链接）

## 输出
到哪去、什么格式、关键字段

## 业务规则
1. （精确、可测试的自然语言描述）
2. ...

## 错误场景
- 条件 → 错误码/行为（标注 [设计行为] 或 [!风险]）

## 可观测状态
- Redis Key / 响应字段 / 日志等可验证的外部状态
- 可观测性盲区（关键路径缺少日志/指标的地方）

## 已有测试覆盖
- 已覆盖：维度列表
- 未覆盖：维度列表

## 关联 L2
- [变更名](../L2/xxx.md) — 一句话说明
```

## 约束

- 每个文件控制在 150 行以内
- 功能差异大的模块拆成独立文件
- 反映当前最新全貌（增量更新时合并变更到已有文件）
- 输入/输出章节：如果系统有 proto 或 OpenAPI 定义，链接到对应定义，不重复列字段
- **接口章节必须记录**：每个有 HTTP/gRPC 测试入口的模块，必须包含完整端点路径和 Schema 链接。即使模块只关注部分字段，也必须至少列出请求体中所有必填字段及其类型（或链接到完整定义）
