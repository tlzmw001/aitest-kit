# 用例格式：共享配置 + 精简用例

> 本文档定义测试用例的 Markdown 输出格式。
> 引用方：test-design skill（生成用例）、codegen（解析 MD 生成 pytest）

## 设计原则

- 每个模块的用例文件顶部定义**共享配置**，包含所有用例共用的部分
- 每条用例只写与共享配置不同的**场景变量**和**断言**
- 目标：人类 review 时一眼看到"这条用例测什么"，不被重复信息干扰

## 文件结构

```markdown
# {模块名} 业务测试用例

> 生成方式：test-design skill
> 关联知识库：L1/xxx
> 生成日期：YYYY-MM-DD

---

## 共享配置

**接口**：`POST /api/v1/recommend` / `gRPC coupon.CouponService/Recommend`

**基础请求体（HTTP）**：
（完整 JSON，用 {{placeholder}} 标记每条用例会替换的字段）

**基础请求体（gRPC）**：
（完整 protobuf message 字段，如有两种接口）

**标准前置**：
- 所有用例共用的环境准备步骤（路由配置、库存初始化等）

**通用断言**：`response.code == 0`

**变量定义**：
- `s` = `response.results[0].score`（打分模型返回的原始分数，不可预知）
- `cal` = `response.results[0].calibrated_score`
- `clamp(x, 0, 1)` = `max(0, min(1, x))`

---

## 一、{场景分组名}

### TC-XXX-001：{一句话描述}
- **优先级**：P1
- **场景变量**：（只写与共享配置不同的部分）
- **断言**：`cal == clamp(1.5 * s + 0.1, 0, 1)`

### TC-XXX-002：{一句话描述}
- **优先级**：P1
- **场景变量**：...
- **断言**：`cal == s`（校准跳过）

---

## 覆盖变更

| 知识库文档 | 新增覆盖 | 仍未覆盖 |
|-----------|---------|---------|
| L1/xxx | 维度 A、维度 B | 维度 C |
```

## 共享配置区字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 接口 | 是 | 列出模块暴露的所有接口类型（HTTP endpoint + gRPC method） |
| 基础请求体 | 是 | 完整的请求 JSON/protobuf，每种接口一份 |
| 标准前置 | 是 | 所有用例共用的环境准备（路由表、库存、基础配置等） |
| 通用断言 | 是 | 每条用例都要检查的断言 |
| 变量定义 | 按需 | 关系断言中的变量缩写，减少每条用例的重复 |

## 每条用例字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 优先级 | 是 | P0/P1/P2，异常类型追加标注（如 `P1 / 异常`） |
| 场景变量 | 是 | 与共享配置的差异：特殊配置、非标准输入、环境变化等 |
| 断言 | 是 | 可程序化的断言表达式，引用变量定义中的缩写 |

## 旧版完整格式（兼容）

已有用例可保留以下格式，不强制迁移：

```markdown
### TC-{模块}-{序号}：{一句话描述}
- **关联**：L1/xxx 或 L2/xxxx
- **优先级**：P0 / P1 / P2
- **类型**：业务 / 边界 / 异常
- **前置条件**：环境状态、数据准备
- **输入**：完整请求体（JSON 格式）
- **测试步骤**：
  1. 动作一
  2. 动作二
- **预期结果**：可断言的具体值
```
