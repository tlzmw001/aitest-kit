# AIAutoTest 协作指南

本文件适用于当前目录及其所有子目录。

它的目标有两层：

1. 说明这个仓库是什么、主要目录做什么、推荐按什么工作流协作
2. 约束 AI 在本仓库中的编码、调试、验证和安全行为

## 项目定位

`AIAutoTest` 是一个 AI 驱动的自动化测试项目，围绕一条稳定的测试飞轮展开：

`开发文档 -> 测试知识库 -> 测试用例 -> 执行 -> 修正与沉淀`

仓库中同时包含：

- 待测系统
- 测试工具
- AI 生成测试资产的工作区

在这个仓库里协作时，优先遵守这条飞轮，而不是直接跳过文档和知识层，直接产出零散测试代码。

## 仓库结构

- `coupon_system/`
  待测系统：智能优惠券推荐策略服务，包含 FastAPI、gRPC 与相关数据层逻辑。

- `docs/`
  开发文档输入目录，通常是文档审查、知识构建、测试设计的起点。

- `test_workspace/`
  AI 生成测试资产的工作目录。

- `test_workspace/knowledge/`
  测试知识库目录，包含 L0/L1/L2 以及 `TEST_SPEC` 等内容。

- `test_workspace/cases/`
  Markdown 测试用例目录。测试用例按模块组织，是后续代码生成或执行前的核心设计产物。

- `test_workspace/plans/`
  测试方案、规划与辅助文档目录。

- `aitest_kit/`
  Python 测试工具库，包含 HTTP/gRPC 客户端与断言能力。

- `.claude/skills/`
  原 Claude 工作流的 skill 定义目录，可作为迁移到 Codex 工作流时的本地 SOP 来源。

## 常用命令

除非用户明确要求别的方式，否则优先使用项目已有命令。

```bash
pip install -e ".[dev,server]"
python -m coupon_system.main
pytest tests/
```

## 测试飞轮工作流

本仓库的核心是测试飞轮。对于中大型测试任务，优先按下面顺序协作：

1. `doc-review`
   审查现有文档，识别缺失、歧义和薄弱点。

2. `doc-gen`
   当文档不足时，从源码补全文档。

3. `knowledge-build`
   从已确认的文档构建或更新测试知识库。

4. `test-design`
   基于知识库和 `TEST_SPEC` 生成测试用例。

5. 人工评审
   生成的测试用例在进入下一阶段前，应允许人工审阅。

6. `test-fix`
   修正错误用例，并把经验沉淀回 `TEST_SPEC` 和相关流程说明中。

如果需求发生变化，默认从 `knowledge-build` 重新进入，除非可以明确证明变化非常局部，且不会影响知识层。

## 推荐使用路径

- 首次接入新项目或新模块：
  先做文档审查，再按需补文档，然后构建知识库，最后设计测试用例。

- 需求迭代：
  将新文档放入 `docs/`，先增量更新知识库，再增量生成或修订测试用例。

- 用例出错或质量不稳定：
  优先修正用例，并把教训沉淀到 `TEST_SPEC` 或相关 workflow 说明里。

- 只检查文档质量：
  只做文档审查，不强制进入后续流程。

## 项目级关键约定

- 测试知识库是测试设计的主输入。
  除非用户明确要求，否则不要绕过知识层，直接从零散文档写测试用例。

- Markdown 测试用例是核心设计产物。
  如果后续存在 codegen 或 pytest 生成环节，应把 Markdown 视为源数据。

- `TEST_SPEC` 是共享行为准则。
  重复踩到的坑、边界条件和经验，应统一沉淀到这里，而不是每次重新摸索。

- 测试用例按模块组织。
  默认输出到 `test_workspace/cases/{模块名}/`。
  若模块不明确，在批量生成前先确认目标模块。

## 如何使用本地 Workflow 文件

`.claude/skills/` 下的文件不会自动作为原生 Codex skill 生效，但它们仍然是有价值的本地流程说明。

当用户明确指定某个本地 skill，或当前任务与其中某个 skill 明显对应时，应按以下方式使用：

1. 阅读对应的 `SKILL.md`
2. 把它作为当前任务的 SOP 执行
3. 保持输出与本仓库的测试飞轮一致

如果后续团队希望把这些流程变成原生 Codex skill，可以再做正式迁移。
