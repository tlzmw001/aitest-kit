# AI Coding Rules

This file governs this directory and all subdirectories.

## 通用行为

- 当我打断或重定向时，立即停止当前方向，按新指令执行
- 当我描述一个具体技术问题时，聚焦该问题本身，不要泛化或扩大范围
- 在开始实际代码编写前（方案设计、技术选型、修复策略阶段），不确定的部分先讨论清楚，不要假设
- 实现范围严格限定在我要求的内容内，不要加额外功能；拿不准就少做
- 调试时如果我提出了假设，先验证我的假设；只有假设被证伪后才扩大搜索范围
- 高风险动作（发布、删除、远程部署）必须先确认目标产物、修改范围、不可触碰项

## 代码规则

- 不要硬编码：端口、URL、配置等全部走环境变量或配置文件
- 禁止过度设计：只做必要功能，不添加额外抽象
- 单文件不超过 500 行：超过必须拆分为独立模块
- 每个修复必须带对应的测试
- 所有已有测试必须通过才能提交
- 不要带任何密钥/敏感信息提交
- 没有数据就显示空白，不要用假数据/示例数据
- 遵循项目已有模式：优先使用项目现有的构建方式，不要绕过去用替代方案
- 禁止擅自修改 .env 等环境配置文件，修改前必须征得我同意
- 禁止静默吞错：禁止对关键 IO/网络调用使用空 catch，所有失败必须记录日志
- 结构变更必须全链路同步：新增字段时同步检查生产代码、测试、序列化、DB 映射

## 修改纪律

- 不修改公开 API 签名，除非我明确要求 breaking change
- 不为只出现 1 次的代码提取抽象（3 行重复好过 1 个过早抽象，等第 3 次再提取）
- 不添加未被要求的功能（bug fix 范围严格锁定，不顺便重构）
- 不在修复中升级依赖版本（依赖升级是独立决策）
- 不删除看起来"没用"的代码而不先确认（可能是进行中的功能）
- 不引入新依赖来解决标准库能解决的问题
- 不在修复中改变代码风格（风格变更独立 commit）
- 不一次性提交多个不相关的修复（原子 commit，方便 revert）
- 进入实际代码编写后，按已确认的目标连续执行；不需要每一步都确认

## 调试协议

修 bug 必须按这个顺序，禁止跳步：

1. **复现** — 先运行命令或写测试，确认 bug 存在（不是看代码猜）
2. **定位根因** — 读错误消息、追踪数据流、对比正常实现
3. **单一假设** — 形成一个假设 → 最小测试 → 验证；不成立则换假设
4. **单一修复** — 只改一个地方，运行测试确认修复 + 无回归
5. **连续 3 次失败 → 停下来**，告诉我卡在哪、已尝试什么、你的假设是什么

## 测试规则

- 测试失败时，修复源代码，不要修改测试来让它通过
- 禁止：放宽断言条件、让验证函数返回 true、skip 失败用例
- 修改测试基础设施（conftest / jest.config / test helpers）必须说明原因
- 新功能优先写失败测试，再写最小实现通过测试
- 测试不依赖极紧时间窗口（如 sleep 50ms 后断言），用事件同步或状态轮询

## 验证规则

声称"已修复"或"已完成"前，必须满足：

1. 给出一条验证命令，证明功能正常
2. 贴出运行结果（不是说"应该可以了"）
3. 构建命令 0 错误
4. 测试命令全部通过

禁止用语（未验证前）：
- "应该可以了"
- "看起来没问题"
- "之前跑过了"
- "理论上已修复"

## 防止重复设计

- 先搜后写：新建文件/类/函数之前，必须先搜索项目中是否已有类似功能
- 已有就优先扩展：搜到类似功能时，优先扩展现有代码；若会导致职责混乱、超过单文件行数上限或破坏模块边界，可新建并说明理由
- 第三次重复时必须抽象：第 1 次直接写，第 2 次容忍重复，第 3 次提取公共代码

## Spec-Driven Workflow（规范驱动开发）

复杂功能（涉及 3 个以上文件）必须先写 spec 再实现：

1. 先理解现有系统 — 读相关代码，画出当前数据流
2. 写 spec — 明确：改什么、为什么改、影响哪些文件、接口定义
3. 按 spec 实现 — 不偏离 spec，发现 spec 有问题先更新 spec
4. 验证 — 实现完成后对照 spec 逐项检查

简单修改（bug fix、单文件改动）不需要 spec，直接做。

## 声明-执行完整性（最容易被忽视的坑）

AI 声明了组件后，必须完成三级验证：

| 级别 | 验证内容 | 示例 |
|------|---------|------|
| L1 存在 | 文件是否已创建 | 声明的模块文件、配置文件 |
| L2 实质 | 是否为真实实现（非占位符） | 不含 todo!()、空函数体 |
| L3 接线 | 是否被正确调用 | Config 被 load()、方法在启动时调用 |

反模式：
- Config 结构体写好了但启动用的是 Default::default() → 配置永远不生效
- save() 方法写好了但从没被调用 → 重启后数据丢失
- 新模块创建了但全是 todo!() → 编译通过但功能缺失

## 安全规则

- 禁止字符串拼接构造 SQL 查询，必须用参数化查询
- 禁止代码中硬编码密钥/凭证，改用环境变量
- 用户输入不能未转义直接输出到 HTML
- API 端点必须有认证/授权检查
- 日志中不能包含密码、token 等敏感信息
- 文件路径必须验证，防止路径遍历
- 安全逻辑必须在业务代码层显式可见，不能只靠框架隐式注入

## 工作流

- 继续执行任务，不用每步都问我确认；仅在以下场景必须先确认：安全高风险操作、修改 `.env` 或其他环境/配置文件、发布/部署、删除/回滚等破坏性操作
- 修改前先完整了解系统，分析后再动手
- 先补可观测性再修逻辑：偶发失败时优先补日志，确认可复现后再修
- 构建失败时必须先修复构建再继续其他编辑，禁止在红灯下新增代码

## 优先级

安全 > 逻辑正确 > 数据一致 > 代码质量 > 性能 > 命名规范
